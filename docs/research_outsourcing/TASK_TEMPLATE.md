# Research Task Template

Copy this block, fill in the bracketed parts, and send it together with
`RESEARCH_BRIEF.md` (as system/context) to the outsourced researcher LM.

---

## Task

**Question:** [one sentence — what specific instrument, fact, or actor are
we trying to find?]

**Why it matters:** [one sentence — which gap/rung/case study does this
close? Cite the doc/section if there is one, e.g. "progress_report_2026-06.md
§5 item 12."]

**What we already know:**
- [Any partial info already in hand — a name, a rough date, a related
  instrument that cites this one, a press article that mentions it without
  the primary text. Give the researcher a running start.]

**What "done" looks like:** [usually: primary text located + captured per
RESEARCH_BRIEF.md §3. Be specific if the bar is different — e.g. "we just
need the instrument number, not the full text" or "we need to know if this
exists at all, a confirmed negative is a valid answer."]

**Known dead ends (don't re-try these):** [optional — if a previous attempt
already ruled out an approach, say so, e.g. "ej.sudrf.ru is confirmed
authenticated-only, don't waste time probing it again."]

**Domains likely to host the answer:** [optional steer — e.g. pravo.gov.ru,

## Reminder for whoever fills this in (delete before sending)

- One task = one question. Don't bundle unrelated instruments into one task;
  it makes the returned report harder to verify and load.
- If the task is "find out whether X exists at all" (a discovery question,
  not a build-ready capture), say so explicitly — a confirmed negative is a
  complete, useful answer and the researcher shouldn't keep digging past a
  legitimate dead end.
- Pull ready-made tasks from `OPEN_QUESTIONS.md` when possible — it's kept in
  sync with the project's own gap register so you're not duplicating
  something already tracked.
